# Contributing to Kiruhi Store

Thanks for helping improve Kiruhi Store.

This project is built to help people discover and install useful apps faster. We care about ideas that solve real needs, reduce friction, and stay aligned with what users are actively searching for.

## Before you start

Please read the README first.

If you plan to work on a new feature, check whether the idea is already covered in an open issue or discussion.

When possible, validate feature ideas with real demand signals such as:

* user feedback
* issue comments
* search interest and product trends
* repeated requests in the community

## Good contribution ideas

The most useful contributions are the ones that improve discovery, trust, speed, or simplicity.

Examples:

* better app search and filtering
* clearer install flow
* safer app verification
* improved GitHub release parsing
* more useful empty states and error messages
* faster loading and smoother UI
* better documentation and onboarding

## What we value

We prefer contributions that are:

* simple
* useful
* easy to maintain
* well tested
* consistent with the project vision

If an idea is flashy but does not improve the user experience, it will probably not be accepted.

## How to contribute

1. Fork the repository.
2. Create a branch for your work.
3. Make your changes.
4. Test everything locally.
5. Open a pull request.

Branch names should be clear, for example:

* `feature/app-search`
* `fix/release-parser`
* `docs/readme-update`

## Pull request rules

Every pull request should include:

* a short summary of what changed
* why the change matters
* screenshots or screen recordings if the UI changed
* steps to test the change

Keep pull requests focused. One PR should do one main thing.

## Code style

* Write clean, readable code.
* Match the existing project style.
* Keep functions small when possible.
* Avoid unnecessary complexity.
* Add comments only when the code is not obvious.

## Testing

Before opening a PR, make sure your change works as expected.

At minimum, test:

* app launch
* search
* app details page
* install or download flow
* error handling

If your change affects logic, add or update tests.

## Documentation

If your change affects how users or contributors work with the project, update the documentation too.

Good docs are part of the feature.

## Security and trust

Do not submit code that:

* downloads unsafe content without warning
* hides what it is doing
* bypasses platform security expectations
* introduces secret tracking or telemetry without clear notice

If you find a security problem, report it privately instead of opening a public issue.

## Reporting bugs

A good bug report includes:

* what you expected
* what actually happened
* steps to reproduce
* device or OS details
* logs or screenshots if available

## Suggesting features

When opening a feature request, explain:

* the problem
* who it helps
* why it matters now
* how it would improve the store experience

Feature requests that match real user demand are much more likely to be accepted.

## Review process

Maintainers may request changes before merging.

Common reasons for changes:

* the idea is unclear
* the PR is too large
* the UI is inconsistent
* the code is hard to maintain
* the feature does not fit the project direction

Please stay responsive during review.

## Community

Be respectful.

Good contributions make the project stronger. Good behavior makes the community worth staying in.

## Need help?

Open an issue or start a discussion with a clear description of what you are trying to build.
